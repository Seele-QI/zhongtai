# 视频提示词板块设计

## 背景
- 当前 `AI 视频创作` 页面在素材准备区存在一块空白区域，适合承载与视频动作控制相关的配置。
- 仓库中的 `数字人44秒_api.json` 已确认 RunningHub 工作流使用 `254` 节点作为文本输入节点，字段为 `254.inputs.text`。
- 该节点当前使用固定多行文案，不支持前端切换模式，也不支持用户手动修改后提交。
- 用户已确认采用方案 1：
- 前端内置三种默认模式。
- 页面首次打开时默认填入“自然模式”。
- 用户点击模式后，对应提示词直接覆盖到提示词板块。
- 用户可以继续手动修改，最终以文本框当前内容提交。

## 目标
- 在素材准备区新增“视频提示词”板块，填充现有空白区域。
- 前端提供三种默认提示词模式，并默认加载“自然模式”。
- 用户点击模式后，可立即将对应 11 行提示词写入文本框。
- 用户可手动编辑提示词，生成时始终提交当前文本框内容。
- 后端在调用 RunningHub 视频工作流时，将当前提示词映射到 `254.inputs.text`。
- 提示词内容和当前模式可随任务状态一起本地持久化，支持切页或刷新后恢复。

## 非目标
- 不将模式模板改为后端动态下发。
- 不新增独立的提示词生成器或 AI 改写能力。
- 不改变现有“图片 + 音频 + 口播文案 + 性别”的主流程。
- 不修改 RunningHub 工作流中除 `254` 节点以外的其他节点结构。

## 方案选择

### 方案 A：前端内置三种模式并直接提交文本框内容
- 模式模板写在前端常量中。
- 默认模式在页面初始化时自动填充到文本框。
- 点击模式按钮后直接覆盖文本框内容。
- 生成提交时把当前文本框内容发给后端，再映射到 RunningHub `254.inputs.text`。
- 结论：采用本方案。

### 方案 B：前端只传模式 ID，由后端返回模板
- 模板更新更灵活，但需要额外查询接口和加载状态。
- 对当前页面来说过重，不符合“点击即填充”的直接交互。

### 方案 C：前后端双份模板兜底
- 容错更高，但维护成本大，容易出现模板不一致。
- 当前需求规模较小，没有必要双份维护。

## 前端设计

### 界面布局
- 文件位置：`components/video-creation-workflow.tsx`
- 在“素材准备”区域的三列布局中，将中间原音频卡片与右侧文案卡片之间的空白位置改为“视频提示词”卡片。
- 为避免挤压现有表单，可将第一步布局调整为更清晰的三栏：
- 左侧：数字人形象。
- 中间：参考音色 + 视频提示词。
- 右侧：口播文案 + 个人名片。
- “视频提示词”卡片结构：
- 标题：`视频提示词`
- 描述：`用于控制数字人口播时的动作节奏，将提交到 RunningHub 254 节点`
- 模式按钮组：`自然模式`、`模式二`、`模式三`
- 多行文本框：用于展示和编辑当前提示词
- 行数建议：11 行左右默认可见，避免用户误以为内容不完整。

### 交互规则
- 页面首次进入且本地不存在已保存提示词时，自动填入“自然模式”。
- 如果本地已存在用户上次编辑过的提示词，则优先恢复本地内容，不强制重置为默认模式。
- 用户点击任一模式按钮后：
- 将该模式的 11 行默认提示词完整写入文本框。
- 更新当前选中模式标记。
- 不弹确认框，保持“点击即生效”的交互。
- 用户在文本框中手动编辑后：
- 不清空当前模式标记。
- 生成时始终提交文本框当前值，而不是模式原始模板。
- 用户删除全部文本后允许继续编辑，但提交时后端需要做空值兜底。

## 状态与持久化设计

### `lib/video-task-store.ts`
- `VideoTaskState` 新增字段：
- `videoPrompt: string`
- `videoPromptMode: "natural" | "mode2" | "mode3"`
- 默认值：
- `videoPrompt` 使用自然模式文本
- `videoPromptMode` 使用 `natural`
- `loadTask()` 兼容旧数据：
- 若历史任务没有这两个字段，则自动补默认值。
- `saveTask()` 与现有文案、名片、音量保持一致，自动持久化这两个字段。

### 前端状态同步
- 页面组件新增便捷别名和 setter：
- `videoPrompt`
- `videoPromptMode`
- `setVideoPrompt`
- `setVideoPromptMode`
- 点击模式按钮时，同时更新两者。
- 用户手动输入文本时，只更新 `videoPrompt`。

## 接口与数据流

### 前端提交
- 文件位置：`components/video-creation-workflow.tsx`
- 生成接口请求体新增字段：
- `video_prompt`
- 可选附带：
- `video_prompt_mode`
- 目的：
- `video_prompt` 负责提交最终使用文本。
- `video_prompt_mode` 主要用于日志、排查和后续扩展，不作为最终内容来源。

### Next API 层
- 文件位置：`app/api/video/generate/route.ts`
- 新增请求参数读取：
- `video_prompt`
- `videoPrompt`
- `video_prompt_mode`
- `videoPromptMode`
- 兼容驼峰和下划线命名，避免前后端过渡期间出错。
- 当 `video_prompt` 为空字符串或纯空白时：
- 自动回退到自然模式模板。
- 再把最终结果写入 RunningHub payload 的 `254.inputs.text`。

### RunningHub 映射
- 目标节点：`254`
- 目标字段：`inputs.text`
- 提交规则：
- 保留换行，不做拆行数组化处理。
- 直接将完整多行字符串写入 `254.inputs.text`。
- 这样可与工作流中后续 `255` 节点的 `easy promptLine` 保持兼容。

## 错误处理
- 用户手动清空提示词：
- 前端允许提交。
- 后端自动回退为自然模式，避免 RunningHub 收到空文本。
- 历史任务缺少新字段：
- `loadTask()` 自动补默认值，不影响旧任务恢复。
- 模式按钮点击后覆盖了用户原内容：
- 这是本次确认的设计行为，不增加确认弹窗。
- RunningHub 返回异常时：
- 仍沿用现有生成失败提示，不额外新增专属错误态。
