# 文案创作板块口播稿纯化与多版本卡片设计

## 背景
- 当前文案创作板块在生成“口播稿”时，模型会输出 `（语气放缓）`、`（停顿）`、`[镜头切换]` 等非口播正文内容。
- 当前展示形态以普通聊天文本为主，长文阅读体验较弱，多版本内容缺少结构化分组。
- 当前“生成视频”操作基于整段 AI 回复，无法保证传递给视频创作板块的内容一定是纯口播稿。

## 目标
- 文案创作板块统一优化为“口播稿优先”输出模式。
- AI 在创作文案时，仅生成可直接朗读的口播稿正文，严禁输出舞台提示、镜头提示、说明文字。
- 当 AI 输出多个版本时，前端按 Markdown 版本标题拆分为独立卡片展示。
- 每个纯口播稿卡片下都提供单独的“跳转视频生成”按钮。
- 跳转到视频创作板块时，粘贴内容必须是纯口播稿，不能包含标题、说明、括号提示等其他内容。

## 非目标
- 不改造视频创作板块的整体流程。
- 不将聊天接口改造成强制 JSON 返回结构。
- 不为非文案创作板块引入新的渲染体系。

## 适用范围
- 作用范围是整个“文案创作”板块，不只限于“高效口播脚本”智能体。
- 板块内 AI 生成结果统一按“口播稿优先”规则约束和渲染。

## 方案选择

### 方案 A：只改提示词
- 仅在系统提示词中禁止舞台提示，并要求按多版本 Markdown 输出。
- 优点：改动小。
- 缺点：模型仍可能偶发输出说明文字；前端无法保证跳转视频时拿到的是纯稿。

### 方案 B：提示词 + 前端结构化切片 + 纯稿跳转
- 在系统提示词中强约束输出格式。
- 前端把 AI 回复按版本标题切成卡片，并对每张卡片做“纯稿净化”。
- “跳转视频生成”只传净化后的纯口播稿。
- 优点：兼顾输出约束、展示体验和跳转可靠性。
- 结论：采用本方案。

### 方案 C：后端强制结构化 JSON
- 优点：结构稳定。
- 缺点：会增加流式聊天复杂度，并破坏现有文本型聊天体验。

## 功能设计

### 1. 输出规则统一收紧
- 文案创作板块统一采用“仅输出可直接朗读的口播稿正文”的规则。
- 明确禁止输出：
- `（语气放缓）`
- `（停顿）`
- `（提高音量）`
- `[镜头切换]`
- `[字幕强调]`
- `适用场景`
- `创作思路`
- `说明`
- `开场说明`
- `总结`
- 当需要输出多个版本时，仅允许使用 Markdown 标题分隔，例如：

```md
## 版本一

这里是纯口播稿正文。

## 版本二

这里是另一版纯口播稿正文。
```

### 2. 多版本卡片展示
- 前端识别 AI 回复中的 Markdown 版本标题，如 `## 版本一`、`## 版本二`、`## 版本三`。
- 若识别到多版本，则拆分成多张独立卡片。
- 若未识别到明确版本标题，则按单卡片模式展示整段内容。
- 每张卡片显示：
- 版本标题
- Markdown 正文内容
- 操作区：`复制纯口播稿`、`跳转视频生成`

### 3. 纯稿净化
- 跳转视频创作前，对每张卡片内容执行纯稿净化。
- 净化规则包括：
- 移除 Markdown 版本标题
- 移除括号/方括号中的舞台提示和镜头提示
- 移除“适用场景”“说明”“创作思路”等说明性段落
- 移除列表符号和无实际朗读价值的标签
- 保留自然段、逗号、句号、问号、感叹号和换行
- 净化结果必须是可直接复制到数字人口播视频里的纯朗读文本。

### 4. 视频生成跳转
- 每张卡片下提供独立的“跳转视频生成”按钮。
- 点击后调用现有 `onJumpToVideo`，但传递值改为净化后的纯口播稿。
- 不再直接把整段原始 AI 回复传给视频创作板块。

## 页面排版设计

### 展示结构
- 聊天历史仍保留原始消息结构，避免破坏当前会话逻辑。
- Assistant 消息在渲染层增加“版本卡片视图”。
- 版本卡片使用清晰的标题、边框、留白和操作区分隔。

### Markdown 优化
- 标题：更明显的字号和颜色层级，用于区分版本。
- 正文：提高行高和段间距，避免大段文字堆积。
- 列表：仅保留轻量样式，防止视觉过载。
- 卡片：圆角、浅边框、柔和阴影，提升阅读感。
- 操作区：固定在卡片底部，按钮语义清晰。

## 数据流设计

### 输入
- 用户发送普通文本到现有 `app/api/ai/chat-stream/route.ts`。

### 生成
- 服务端基于文案创作板块规则拼装更严格的系统提示词。

### 渲染
- 前端收到 assistant 消息文本后：
- 先保留原始消息到聊天会话历史
- 再解析版本结构
- 最终以单卡片或多卡片方式渲染

### 跳转
- 用户点击某一版本卡片下的“跳转视频生成”
- 对该卡片正文执行纯稿净化
- 将净化后文本传入视频创作板块

## 代码落点

### `lib/prompts/copywriting-agent-systems.ts`
- 为文案创作智能体增加统一的“纯口播稿”输出规则。
- 删除或禁止当前允许的轻量舞台提示描述。
- 明确多版本只允许使用 Markdown 标题分隔。

### `app/api/ai/chat-stream/route.ts`
- 在构建系统提示词时，确保文案创作场景统一附加“纯口播稿”全局规则。
- 不改变现有流式响应协议。

### `components/copywriting-chat-workspace.tsx`
- 新增版本解析函数：将 assistant 文本按版本标题拆段。
- 新增纯稿净化函数：提取可直接用于视频生成的纯正文。
- 优化 Markdown 渲染组件样式。
- 在每个版本卡片下新增：
- `复制纯口播稿`
- `跳转视频生成`

## 错误处理
- 若 AI 未按规范输出多版本标题，则回退为单卡片模式。
- 若净化后文本为空，则禁用“跳转视频生成”并提示用户重新生成。
- 若用户未配置视频跳转能力，则隐藏或禁用该按钮，保持现有容错逻辑。

## 验证方案
- 输入“写 3 个版本的装修避坑口播稿”，页面应拆成 3 张版本卡片。
- 若 AI 回复中混入 `（语气放缓）` 等提示，点击“跳转视频生成”后，视频创作输入框中不应出现这些内容。
- 单版本输出时，应保持单卡片展示，并正常复制与跳转。
- Markdown 标题、段落、列表在新样式下可读性明显优于当前普通聊天文本。

## 风险与控制
- 风险：全板块统一“口播稿优先”后，部分非口播类文案需求可能显得过于收束。
- 控制：先以当前用户明确诉求为准实施；若后续发现影响其他文案场景，再增加“口播模式/普通模式”切换。

## 实施建议
- 先落地提示词与前端解析/净化/卡片渲染。
- 保持接口协议不变，降低改动范围。
- 完成后使用真实样例验证“版本拆分”和“视频跳转纯稿”链路。
