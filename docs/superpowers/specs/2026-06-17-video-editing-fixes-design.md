# 视频剪辑问题修复设计（方案 B）

## 背景
- 当前项目的视频后处理同时服务于两条链路：
- 自动链路：数字人视频生成完成后自动执行后处理。
- 手动链路：用户上传现成视频后执行剪辑与字幕烧录。
- 当前剪辑效果存在 5 个明确问题：
- 没有稳定插入背景音乐 BGM。
- 最终视频时长与口播音频时长不一致。
- 字幕出现时间依赖字符权重估算，和真实语音节奏存在偏差。
- 字幕内容中仍会出现 `/`、`，`、`。` 等符号。
- 剪辑过程中不应调节画面滤镜。

## 目标
- 自动链路与手动链路统一修复 BGM 缺失问题。
- 最终输出视频时长严格对齐主音轨时长。
- 在不引入 ASR 的前提下，优化字幕时间轴算法，使字幕同步明显优于当前实现。
- 字幕烧录结果仅保留文字、数字、英文字母与必要空格，不保留标点和特殊字符。
- 明确限制视频剪辑过程只处理字幕、名片和音频，不调节画面滤镜。

## 非目标
- 不引入 ASR、Whisper、云端语音识别或逐词对齐能力。
- 不改造 RunningHub 视频生成主链路本身。
- 不新增独立的视频编辑器或重构前端剪辑界面。
- 不改造 Remotion 备用剪辑链路，本次只修复当前真实在用的 Python + FFmpeg 后处理链路。

## 方案选择

### 方案 A：引入 ASR 时间戳对齐
- 优点：字幕同步最准确。
- 缺点：需要新增识别依赖、环境变量和失败回退逻辑，超出当前用户确认范围。

### 方案 B：修复 BGM、时长、字幕清洗，并改良字符权重算法
- 优点：改动集中在现有后处理模块，风险可控，能同时覆盖自动链路和手动链路。
- 缺点：字幕同步只能做到“明显更合理”，不能达到真实语音时间戳级精度。
- 结论：采用本方案。

### 方案 C：重做为本地离线 ASR + 强制对齐
- 优点：理论上无需外部服务。
- 缺点：现有项目没有对应依赖，接入成本高，部署复杂。

## 适用范围
- 自动后处理链路：`main.py` 中 `_run_post_process()` 触发的剪辑流程。
- 手动剪辑链路：`POST /api/video/edit` 和 `POST /api/video/manual-edit` 触发的剪辑流程。
- 核心实现模块：`lib/video_postprocess.py`。

## 功能设计

### 1. BGM 修复
- 自动后处理链路当前只传入 `task_id / input_path / output_dir / script / keep_original`，没有把 `bgm_dir / preset / bgm_volume` 传入底层函数。
- 修复后，自动链路与手动链路都通过统一参数入口决定是否混入 BGM。
- 当存在可用 BGM 目录且音量大于 0 时：
- 必须选中一首真实 BGM 文件。
- 必须在 FFmpeg 命令中生成第二路音频输入并执行 `amix`。
- 必须将 BGM 裁切到目标时长，并在尾部做淡出。
- 当未配置 BGM 或目录为空时：
- 仍允许剪辑继续执行，但应明确记录“本次未混入 BGM”的原因，避免误判为功能异常。

### 2. 视频时长修复
- 当前后处理时长基准来自输入视频中的音频流时长，不一定可靠反映最终口播主音轨时长。
- 修复后统一采用“主音轨目标时长”作为后处理全局时长基准。
- 具体策略：
- 优先使用输入媒体第一路音频流时长。
- 若音频流时长不可读，则退回容器总时长。
- 所有音频处理和最终输出裁切都使用同一个目标时长值。
- 最终 FFmpeg 输出必须强制截断到目标时长，不允许输出比主音轨更长的视频。
- BGM 也必须被裁切或填充到同一目标时长，但不能反向拉长最终视频时长。

### 3. 字幕对齐算法优化
- 本次不引入 ASR，保留“按脚本内容推算时间轴”的设计，但改进当前纯字符平均分配逻辑。
- 新算法原则：
- 先对脚本按自然段、停顿符、长短句做稳定分段。
- 每个片段根据字符数、词长、停顿符权重共同计算时长占比。
- 为短句设置最小时长，避免一闪而过。
- 为超长句设置上限，避免单句占据过长时间。
- 对首句、尾句增加缓冲补偿，使开头和结尾更自然。
- 最终所有分段累计时长必须严格等于目标音频时长。
- 若清洗后字幕段为空，则直接丢弃该段，并把剩余时长重新分配给有效字幕段。

### 4. 字幕字符清洗
- 字幕在进入时间轴和 ASS 生成前统一做严格净化。
- 净化后只允许保留：
- 中文
- 英文字母
- 数字
- 必要空格
- 明确移除：
- `/`
- `，`
- `。`
- `！`
- `？`
- `：`
- `；`
- `、`
- 引号
- 括号
- 书名号
- 破折号
- 省略号
- 其他特殊符号
- 若某一字幕片段清洗后为空，则不生成该条字幕事件。

### 5. 禁止画面滤镜
- 本次修复明确规定：后处理不允许使用任何画面调色、磨皮、模糊、增强类滤镜。
- 允许的视频滤镜仅包括：
- `subtitles`
- 用于业务名片叠加的 `drawtext`
- 不允许出现：
- `eq`
- `curves`
- `hue`
- `boxblur`
- `unsharp`
- `vibrance`
- 其他任何改变画面视觉风格的 FFmpeg 视频滤镜

## 数据流设计

### 自动链路
- RunningHub 返回原始视频 URL。
- 后端下载原始视频到本地临时目录。
- 后端统一构造后处理参数，包含字幕文本、BGM 参数、预设参数、时长基准。
- `run_ffmpeg_post_process()` 根据目标时长完成字幕文件生成、音频混音和最终导出。
- 导出成功后更新任务状态为 `published`，并回写后处理视频地址。

### 手动链路
- 用户上传视频，或传入 `video_url / video_base64 / upload_id`。
- 后端统一转成本地输入视频路径。
- 后端复用与自动链路完全一致的后处理入口。
- 剪辑完成后输出静态可访问的视频地址。

## 代码落点

### `lib/video_postprocess.py`
- 重构目标时长计算逻辑，确保字幕、音频、最终输出使用同一时长基准。
- 改造字幕清洗逻辑，只保留文字数字英文和必要空格。
- 重写或增强字幕时间轴分配函数，提升字幕与口播节奏的一致性。
- 显式限制 FFmpeg 视频滤镜链，只允许 `subtitles` 和 `drawtext`。
- 在 BGM 混音逻辑中补充更清晰的命令生成与调试日志输出。

### `main.py`
- 修改 `_run_post_process()`，让自动后处理链路也能传入完整的 BGM 和预设参数。
- 保持 `_run_edit_job()` 与自动链路使用统一的后处理参数约定。
- 若需要，给 `_task_store` 增加最小必要字段，以便自动后处理知道当前应使用的 BGM 目录、预设和音量。

## 错误处理
- 当 BGM 目录不存在或目录为空时：
- 剪辑任务不失败，但在结果日志中记录“未找到可用 BGM”。
- 当字幕清洗后全部为空时：
- 剪辑任务失败，并返回明确错误，提示字幕文本不可用。
- 当音频时长探测失败时：
- 回退到容器时长，但写出调试日志。
- 当 FFmpeg 执行失败时：
- 保留命令文件和错误日志文件，便于定位是字幕、音频还是输入媒体问题。

## 验证方案

### 自动化验证
- 为 `lib/video_postprocess.py` 增加针对纯函数和命令拼装逻辑的测试。
- 覆盖用例：
- 字幕清洗后不包含 `/`、`，`、`。` 等符号。
- 时间轴总时长严格等于目标时长。
- 开启 BGM 时命令中包含第二路音频输入和 `amix`。
- 视频滤镜链不包含任何画面滤镜。

### 手工回归
- 自动链路：生成一段带口播音频的视频，确认后处理结果包含 BGM。
- 手动链路：上传一个带音频的视频，确认输出视频时长和原主音轨一致。
- 使用包含大量标点的脚本，确认字幕成片中只保留文字。
- 对比修复前后同一脚本，确认字幕节奏明显更接近实际口播。

## 验收标准
- 自动链路与手动链路都能稳定混入 BGM。
- 最终视频时长与目标主音轨时长一致。
- 字幕不包含 `/`、`，`、`。` 等标点和特殊字符。
- 字幕同步效果明显优于当前版本。
- 剪辑输出过程中不对画面做滤镜调节。

## 风险与控制
- 风险：不引入 ASR 的前提下，字幕同步仍然无法达到逐句真实时间戳精度。
- 控制：本次明确目标是“显著改良”，不是“逐字精确对齐”；若后续仍不满足，再升级为 ASR 方案。
- 风险：自动链路增加 BGM 参数后，可能和历史任务状态兼容性产生差异。
- 控制：为缺省值提供稳定默认行为，未配置时仍允许无 BGM 成功出片。

## 实施建议
- 第一步：梳理并统一后处理参数入口。
- 第二步：修复自动链路 BGM 透传与目标时长基准。
- 第三步：改造字幕清洗与时间轴算法。
- 第四步：补测试与真实样例回归。
